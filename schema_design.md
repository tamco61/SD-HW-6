# Проектирование документной модели для MongoDB

В рамках миграции и развития системы BlablaCar часть сущностей была перенесена из реляционной СУБД (PostgreSQL) в документоориентированную СУБД (MongoDB). Ниже описана и обоснована спроектированная структура коллекций.

## 1. Коллекция `users` (Пользователи)

Хранит профили пользователей системы. Хотя пользователи могут существовать и в PostgreSQL, мы дублируем/переносим базовую информацию для демонстрации связей (references) в MongoDB.

**Структура документа:**
```json
{
  "_id": "ivan_petrov",
  "first_name": "Ivan",
  "last_name": "Petrov",
  "password_hash": "hash_string",
  "created_at": ISODate("2023-10-01T12:00:00Z")
}
```
*В качестве `_id` используется уникальный `login` пользователя.*

## 2. Коллекция `trips` (Поездки и маршруты)

В реляционной модели (домашнее задание №3) маршруты (`routes`) и поездки (`trips`) были разделены на разные таблицы. В MongoDB мы объединяем их для оптимизации частых запросов (поиск поездок всегда требует информации о маршруте).

**Структура документа:**
```json
{
  "_id": ObjectId("651ef1..."),
  "owner_login": "ivan_petrov",
  "route": {
    "points": ["Moscow", "Kazan", "Yekaterinburg"],
    "distance_km": 1500
  },
  "departure_time": ISODate("2023-11-01T08:00:00Z"),
  "status": "active",
  "participants": [
    "sergey_novikov",
    "maria_ivanova"
  ],
  "created_at": ISODate("2023-10-05T10:00:00Z")
}
```

## 3. Обоснование выбора между Embedded (Встраивание) и References (Ссылки)

### Маршрут (`route`): Embedded (Встраивание)
- **Причина**: В контексте нашей системы каждый созданный маршрут жестко привязан к конкретной поездке. Маршрут не переиспользуется множеством различных поездок независимо (чаще всего водитель просто создает новую поездку и указывает путь).
- **Преимущества**: При поиске поездок или их отображении на главной странице клиенту всегда нужны точки маршрута. Встраивание (embedding) объекта `route` напрямую в документ `trip` позволяет получать всю необходимую информацию одним запросом к базе (без `$lookup`), что значительно повышает производительность на чтение.

### Владелец (`owner_login`): Reference (Ссылка)
- **Причина**: Владелец является полноправной сущностью "Пользователь". Профиль пользователя может меняться (смена фамилии, настроек), и он может владеть множеством поездок.
- **Преимущества**: Хранение только идентификатора (`login`) позволяет избежать каскадных обновлений (update anomalies). При необходимости вывести имя водителя, мы делаем `$lookup` к коллекции `users`.

### Участники (`participants`): Reference (Массив ссылок)
- **Причина**: Пассажиры — это независимые пользователи системы. Кроме того, их количество в рамках одной машины обычно невелико (до 4-8 человек), поэтому массив идентификаторов (`login`) не приведет к превышению лимита размера документа MongoDB (16 MB).
- **Преимущества**: Как и с владельцем, профили участников хранятся в одном месте (в `users`). Если пользователь меняет свое имя, оно автоматически будет отображаться корректно во всех его поездках после `Join`/`$lookup`. Паттерн *Array of References* идеально подходит для связи "Многие ко Многим" (Many-to-Many), когда размер массива ограничен бизнес-логикой.
